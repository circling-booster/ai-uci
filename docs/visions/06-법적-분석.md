# 법적 분석 — Local-First 구조의 법적 책임

> 작성일: 2026-03-01  
> 상태: 초기 분석 (정식 법률 자문 필요)

---

## 핵심 전제

> Phase 1에서는 중앙 서버 없이, **사용자 본인의 디바이스에서 본인의 브라우저/계정으로**  
> 플랫폼에 접근한다. 이 구조가 법적 책임에 미치는 영향을 분석한다.

---

## 1. 주요 법적 쟁점별 분석

### 1.1 이용약관(ToS) 위반

| 항목 | 중앙 서버형 | Local-First (Phase 1) |
|------|-----------|----------------------|
| 위반 주체 | **운영자** (서버가 직접 접근) | **사용자 개인** (본인 브라우저) |
| 운영자 책임 | 직접 책임 | **도구 제공자로서의 간접 책임** |
| 위반 강도 | 높음 (대량/체계적 접근) | 낮음 (개인적/간헐적 접근) |

**분석**:
- 사용자가 본인 계정으로 본인 브라우저를 통해 접근하므로, ToS 위반 주체는 **사용자 개인**
- 그러나 도구 제공자도 **기여 침해(contributory infringement)** 또는 **유인 침해(inducement)** 책임을 질 수 있음
- **완화 방법**: ToS 준수 안내, 사용자 동의(면책 조항), 플랫폼 공식 API 우선 사용

### 1.2 저작권 침해

| 항목 | 분석 |
|------|------|
| 콘텐츠 복제 | 사용자 로컬에 캐시/저장 → **사적 이용(private use)** 항변 가능 |
| 재배포 | 로컬에서만 사용, 외부 전송 없음 → 재배포에 해당하지 않음 |
| 변환적 사용 | LLM 요약/분석은 **transformative use**로 볼 여지 |
| 운영자 역할 | 콘텐츠를 직접 호스팅/배포하지 않음 → 직접 침해 없음 |

**분석**:
- 사용자가 본인 디바이스에서 콘텐츠를 보는 것은 **일반적인 브라우저 사용과 본질적으로 동일**
- 핵심 구분: 도구가 콘텐츠를 **재호스팅/재배포하는가?** → No (Local-First이므로)
- **주의**: 동영상 스트림 직접 추출(yt-dlp 방식)은 저작권법 위반 리스크 높음 → embed/iframe 방식이 안전

### 1.3 컴퓨터 사기 및 남용법 (CFAA / 한국 정보통신망법)

| 법률 | 적용 가능성 | 분석 |
|------|------------|------|
| CFAA (미국) | 낮음 | 사용자 본인 계정으로 접근, "무단 접근" 성립 어려움 |
| 정보통신망법 (한국) | 낮음 | 정당한 접근 권한(본인 계정) 보유, 자동화 자체를 금지하지 않음 |
| 부정경쟁방지법 (한국) | 중간 | "상당한 투자·노력으로 만든 성과"를 무단 사용하는 경우 해당 가능 |

### 1.4 도구 제공자로서의 책임

**핵심 질문**: "이 도구를 만들어 배포한 것 자체가 위법한가?"

| 법리 | 내용 | 적용 |
|------|------|------|
| **Sony Betamax 판결 (1984)** | 합법적 용도가 있는 기술의 제조·배포는 적법 | ✅ 유리 — 공개 정보 열람은 합법적 용도 |
| **MGM vs Grokster (2005)** | 침해를 유인·장려하면 책임 있음 | ⚠️ 마케팅/문서에서 "ToS 우회" 등을 장려하면 위험 |
| **기여 침해 (Contributory)** | 침해 사실을 알면서 실질적 기여를 한 경우 | ⚠️ 구조적으로 침해를 방지하는 설계가 필요 |

## 2. Local-First 구조의 법적 이점

### ✅ 유리한 요소

1. **운영자가 콘텐츠를 수집·저장·배포하지 않음**
   - 직접적 저작권 침해 성립 불가
   - DMCA 등의 직접 대상이 되지 않음

2. **사용자 본인의 브라우저/계정 사용**
   - 정당한 접근 권한 보유
   - "무단 접근" 주장이 어려움
   - 일반 브라우저 사용과 기술적으로 동일

3. **데이터가 사용자 로컬에만 존재**
   - 개인정보보호법(GDPR, 개인정보보호법) 의무 최소화
   - 데이터 유출 리스크 제거
   - 사용자가 자신의 데이터 완전 통제

4. **오픈소스 배포 시 추가 이점**
   - 코드 투명성 → 악의적 용도 아님을 입증 용이
   - 커뮤니티 검증
   - 참고: VLC, yt-dlp 등 유사 도구가 오픈소스로 유지되는 선례

### ⚠️ 여전히 존재하는 리스크

1. **유인 침해 (Inducement)**
   - 도구의 주목적이 ToS 위반을 조장하는 것으로 보일 경우
   - **완화**: 합법적 용도를 명확히 하고, ToS 준수를 권고

2. **기술적 보호조치 우회 (DMCA §1201 / 한국 저작권법 §104조의2)**
   - DRM 등 기술적 보호조치를 우회하면 별도 위법
   - **완화**: DRM 콘텐츠에는 접근하지 않는 설계

3. **부정경쟁방지법 (한국)**
   - 플랫폼의 "상당한 투자로 만든 성과"를 무단 이용하는 경우
   - **완화**: 공식 API 우선, 공개 데이터만 접근

4. **플랫폼의 기술적 대응**
   - bot 탐지 강화, API 제한, IP 차단 등
   - 법적이 아닌 기술적 제약으로 서비스 불가능해질 수 있음

## 3. 법적 리스크 완화 설계 원칙

### 필수 적용 사항

```
[1] 면책 조항 (Disclaimer)
    - 사용자에게 각 플랫폼 ToS 준수 의무를 명확히 고지
    - "이 도구의 사용으로 인한 ToS 위반은 사용자 책임"

[2] 공식 API 우선 정책
    - 가능한 경우 플랫폼 공식 API 사용
    - API 미제공 기능에 한해 브라우저 자동화 사용

[3] DRM/기술적 보호조치 비우회
    - DRM 콘텐츠 접근 시도하지 않음
    - 기술적 보호조치 우회 기능 미탑재

[4] 콘텐츠 미재배포
    - 수집 콘텐츠는 사용자 로컬에만 존재
    - 외부 서버 전송/공유 기능 미탑재

[5] Rate Limiting / 예의 있는 접근
    - 과도한 요청 방지 (요청 간 딜레이)
    - robots.txt 존중 옵션 제공

[6] 합법적 용도 명시
    - 문서/마케팅에서 합법적 활용 사례 강조
    - ToS 우회, 광고 차단 등을 주요 기능으로 홍보하지 않음
```

### 앱 내 사용자 동의 흐름

```
[최초 실행]
    ↓
[이용 약관 표시]
  - "이 도구는 사용자의 편의를 위한 브라우저 자동화 도구입니다"
  - "각 플랫폼의 이용약관을 준수할 책임은 사용자에게 있습니다"
  - "이 도구는 저작물을 복제·재배포하지 않습니다"
    ↓
[사용자 동의]
    ↓
[플랫폼별 ToS 요약 표시 (선택적)]
  - "YouTube: 자동화된 접근은 ToS에 의해 제한될 수 있습니다"
    ↓
[사용 시작]
```

## 4. 참고 선례 (유사 프로젝트)

| 프로젝트 | 형태 | 법적 상태 | 시사점 |
|----------|------|----------|--------|
| **yt-dlp** | 오픈소스 CLI 도구 | 운영 중 (youtube-dl 후속) | 도구 자체보다 사용 방식이 쟁점 |
| **Stremio** | 로컬 앱 + 애드온 | 운영 중 | 플랫폼과 콘텐츠 접근을 분리 |
| **Invidious** | 유튜브 대안 프론트엔드 | 법적 압박 경험 | **중앙 서버형은 리스크 높음** |
| **NewPipe** | 안드로이드 로컬 앱 | 운영 중 | 로컬 실행 + 오픈소스 모델 |
| **Nitter** | 트위터 대안 프론트엔드 | 서비스 중단 | **중앙 서버형 → API 차단으로 불가** |
| **RSS Bridge** | 오픈소스, 셀프호스팅 | 운영 중 | 사용자 셀프호스팅 = 책임 분산 |

**교훈**:
- 🔴 중앙 서버형 (Invidious, Nitter): 법적 압박 + 기술적 차단에 취약
- 🟢 로컬 실행형 (yt-dlp, NewPipe, Stremio): 상대적으로 안전하게 운영 중
- → **Local-First 결정은 법적 관점에서도 올바른 방향**
