```
[Phase 0] AgentHub 마무리
    ↓
[Phase 1] 콘텐츠 규격화 (Content Schema)
```

## Phase 1: 콘텐츠 규격화 (Content Schema)

### 목표

전 세계 주요 사이트들의 정보 제공 방식을 분석·취합하여, **콘텐츠 유형별 표준 스키마**를 정의한다.

### 콘텐츠 유형별 규격 예시

#### 1) 동영상 콘텐츠

```yaml
ContentType: Video
Schema:
  metadata:
    title: string
    creator: Creator
    duration: Duration
    published_at: DateTime
    platform_source: string        # youtube, vimeo, etc.
    tags: string[]
    category: string
  media:
    stream_url: string             # 실제 스트리밍 소스
    thumbnail: Image
    quality_options: QualityOption[]
    subtitles: Subtitle[]
  social:
    comments: Comment[]            # 규격화된 댓글 구조
    reactions: Reaction[]          # 좋아요, 싫어요 등
    view_count: number
    share_count: number
  related:
    related_videos: VideoRef[]
    related_articles: ArticleRef[]
    trending_context: TrendingInfo[]  # 관련 실시간 이슈
```

#### 2) 기사/뉴스 콘텐츠

```yaml
ContentType: Article
Schema:
  metadata:
    headline: string
    subheadline: string
    author: Author
    publisher: Publisher
    published_at: DateTime
    updated_at: DateTime
    category: string
    tags: string[]
  body:
    lead: string                   # 리드 문단
    sections: Section[]            # 본문 섹션들
    media: MediaEmbed[]            # 삽입 이미지/동영상
    quotes: Quote[]                # 인용문
  social:
    comments: Comment[]
    reactions: Reaction[]
  context:
    related_articles: ArticleRef[]
    fact_check: FactCheckInfo[]    # LLM 기반 팩트체크 정보
    timeline: TimelineEvent[]      # 관련 사건 타임라인
```

#### 3) 댓글 (공통 규격)

```yaml
ContentType: Comment
Schema:
  author: 
    display_name: string
    avatar: Image
    platform_id: string
  content:
    text: string
    media: MediaEmbed[]            # 댓글 내 이미지/GIF 등
    mentions: Mention[]
  metadata:
    posted_at: DateTime
    reactions: Reaction[]
    reply_count: number
  replies: Comment[]               # 대댓글 (재귀 구조)
  ai_analysis:                     # LLM 개입 영역
    sentiment: string              # 긍정/부정/중립
    summary: string                # 댓글 요약 (대량일 경우)
    relevance_score: number        # 관련성 점수
```

### LLM 개입이 필요한 영역

| 단계 | LLM 역할 | 설명 |
|------|----------|------|
| **수집** | 의도 해석 | 사용자의 자연어 요청을 구체적 검색 쿼리로 변환 |
| **수집** | 소스 선택 | 어떤 플랫폼에서 정보를 가져올지 판단 |
| **변환** | 구조 인식 | 비정형 웹 콘텐츠에서 규격 필드 추출 |
| **변환** | 댓글 분석 | 대량 댓글의 요약, 감정 분석, 필터링 |
| **제공** | 관련 정보 연결 | 콘텐츠와 관련된 이슈/기사/트렌드 연결 |
| **제공** | 모호성 해소 | 후보가 다수일 때 사용자에게 적절히 되묻기 |
| **제공** | 맞춤화 | 사용자 선호에 따른 정보 필터링/재배치 |

## Phase 2: 수집·변환 엔진

### 아키텍처 구상

```
[사용자 요청 (자연어)]
        ↓
[LLM Intent Parser] — 의도 분석 및 쿼리 생성
        ↓
[Source Router] — 적합한 플랫폼/소스 선택
        ↓
[Content Extractor] — 브라우저 자동화(CDP) / API 호출
        ↓
[Content Normalizer] — 규격화 스키마에 맞게 변환
        ↓
[Context Enricher] — 관련 정보 연결 (LLM 활용)
        ↓
[Personalization Engine] — 사용자 설정에 따른 필터/배치
        ↓
[Unified UI Renderer] — 사용자 맞춤 인터페이스로 렌더링
```

## Phase 3: 통합 사용자 인터페이스

### 사용 시나리오 예시

#### 시나리오 1: 동영상 시청

> 사용자: "xxx 동영상 틀어줘"

1. LLM이 의도 해석
2. 사용자 설정에 따른 소스 선택 (유튜브 우선 / 기타 플랫폼)
3. 검색 수행 (사용자에게 비노출)
4. 후보가 불명확할 경우 → 사용자에게 되묻기
5. 동영상 스트리밍 + 규격화된 댓글/대댓글 동시 표시
6. 관련 이슈 (기사, 나무위키, 실시간 검색 등) 패널 제공
7. 모든 UI 요소는 사용자 맞춤 스타일 적용

#### 시나리오 2: 뉴스 읽기

> 사용자: "오늘 주요 뉴스 보여줘"

1. 다수 뉴스 소스에서 기사 수집
2. LLM이 중요도/관련성 기준으로 큐레이션
3. 규격화된 형태로 헤드라인 → 요약 → 전문 계층 제공
4. 관련 기사 간 타임라인/맥락 연결
5. 댓글/여론 요약 함께 제공

### 사용자 맞춤화 설정 요소

```yaml
UserPreferences:
  display:
    theme: dark | light | custom
    font_size: number
    layout: compact | comfortable | spacious
  content:
    show_comments: boolean
    comment_style: { color, font, position }
    show_ads: boolean                        # 기본 false
    show_related: boolean
    auto_translate: boolean
  sources:
    preferred_video: [youtube, vimeo, ...]
    preferred_news: [reuters, bbc, ...]
    blocked_sources: string[]
  ai:
    auto_summarize: boolean
    sentiment_analysis: boolean
    fact_check: boolean
    clarification_threshold: number          # 얼마나 적극적으로 되물을지
```

## Phase 4: 제품 형태

### 후보 형태

| 형태 | 장점 | 단점 |
|------|------|------|
| **데스크톱 앱** (Electron/Tauri) | 브라우저 엔진 내장 가능, CDP 활용 용이 | 설치 필요, 모바일 커버 불가 |
| **모바일 앱** | 접근성 높음, 상시 사용 가능 | 브라우저 자동화 제약 |
| **브라우저 확장** | 설치 간편, 기존 브라우징과 통합 | 기능 제약, 스토어 정책 |
| **웹 앱 + 백엔드** | 크로스플랫폼, 설치 불필요 | 서버 비용, 지연 시간 |
| **하이브리드** | 유연성 극대화 | 개발 복잡도 증가 |

> **결정**: 초기(Phase 1)에는 **데스크톱 앱(Tauri) + Local Engine** (중앙 서버 없음)  
> 사용자 디바이스에서 모든 처리 수행, 모바일은 로컬 네트워크 연동
