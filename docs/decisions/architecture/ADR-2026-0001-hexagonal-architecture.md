# ADR-2026-0001: 헥사고날 아키텍처 전환

## 1) 배경
- 기존 계층형 구조에서는 내부 모듈이 인프라 세부 구현에 과도하게 의존해, 비즈니스 규칙 변경 시 테스트 비용과 배포 리스크가 커짐.
- 운영 규칙 변경/어댑터 추가가 잦아 의존성 방향 제어가 어려웠고, 핵심 도메인 유스케이스를 안정적으로 보존하기 어려움.

## 2) 결정 필요성
- 도메인 중심 설계가 명확해야 하며, 외부 기술 의존을 경계하기 위해 중심부(도메인)와 주변(어댑터/포트) 분리 규칙을 강제해야 함.
- 장기적으로 플랫폼 교체, 저장소 교체, 외부 API 통합 추가 시도 비용을 줄여야 함.

## 3) 대안
### 대안 A: 계층형 유지 + 가드 레이어 추가
- 장점: 전환 비용 최소
- 단점: 의존성 방향이 완전하게 정리되지 않고 경계가 약함

### 대안 B: 헥사고날 아키텍처 도입
- 장점: 도메인-포트 분리, 어댑터 독립성, 테스트 대상 분리 용이
- 단점: 초기 설계 비용 증가, 도메인 경계 모델링 노력 필요

### 대안 C: 마이크로서비스 분리로 해결
- 장점: 배포 단위 분리로 독립성 증대
- 단점: 운영 복잡도 상승, 초기 과도한 통신 비용

## 4) 결정
- 본 프로젝트는 **헥사고날 아키텍처(Ports and Adapters)**를 채택한다.
- 핵심 규칙
  - 비즈니스 유스케이스/애플리케이션 서비스는 외부 인프라(저장소, API, UI Adapter)에 직접 의존하지 않는다.
  - 도메인은 Output Port와 Input Port를 통해서만 외부 계층과 상호작용한다.
  - 구현체는 `infrastructure` 또는 `adapter` 계층에서 제공한다.

## 5) 영향도
### 기대 효과
- 핵심 정책(도메인)의 변경 회귀를 줄이고, 테스트용 퍼셉션을 확보한다.
- 인터페이스 표준화로 신규 연동 어댑터 추가 비용을 낮춘다.

### 위험/트레이드오프
- 초기 설계/리팩터링 비용이 발생한다.
- 팀의 포트/어댑터 네이밍 일관성 규칙이 필요하다.
